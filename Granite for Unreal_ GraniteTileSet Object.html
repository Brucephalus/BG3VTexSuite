Granite for Unreal adds a new object type to unreal the GraniteTileSet. Together with the GraniteTexture object this is one of the key objects of Granite for Unreal. A GraniteTileSet object represents a Tile Set within the unreal engine. It exposes settings specific to the use of tile sets within unreal.

Creating a Granite Tile Set

A GraniteTileSet object is created by importing an ".gts" file using the standard unreal import functionality. See Tile Set Import Factory for more information on the ".gts" importer.

GraniteTileSet Editor

The GraniteTileSet editor is opened by editing a GraniteTileSet from within the unreal editor (e.g. by double clicking it in the asset browser). It exposes the following options.

  • Lod Bias: Level of detail resolution offset. See the Level of Detail page for a detailed discussion of this important parameter.
  • Transcoder Quality: When decoding compressed data Granite can trade off quality for processing time. By lowering the transcoder quality data will be processed faster (i.e. Granite will use less CPU time) but visual quality may be reduced (i.e. more visible "coding artefacts".)
  • Number of Mips to Prefetch: When initially starting the level Granite does not load any texture data. It relies entirely on it's fast streaming to load texture data after the level was loaded. If a lot of data is visible on screen at once, this loading process may be noticeable to the user. To avoid this, Granite has the option to prefetch the lowest n mipmap-levels during the level load. This will increase level loading times but reduce visible loading the first few frames after the level started. Setting this to a value like 4 or 5 will give good results for most usuage scenarios giving a good balance between loading times and visible streaming.
  • Number of Mips to Pin: With prefetching the data is guaranteed to be available when the project starts up. However Even during the course of the game this data may still be removed from memory if it hasn't been visible for a while. Pinning data in memory ensures it will always be resident. Note this options should be used sparingly as it essentially overrides many of the smart decisions Granite tries to make. However, it may help with camera cuts accross areas or scenes where there is little or no data resident on the other side of the cut (as this data hasn't been vsisible for a long time, if ever). Ideally you should set this to a low setting in the range of 1-4. You also likely want to set it equal or lower than the prefetch value. This will ensure initially data is available but during the course of the Game Granite has additional freedom to chose which data to keep in memory.
  • Gpu Cache Group: The GPU cache group the tile set should use. This impacts the type of GPU cache created for this tile set. You can consult the Cache Settings to see which tile set layer uses which type of GPU cache. Currently there are two groups:
    • Standard: All layers of this tile set will use GPU caches with compressed texture formats (e.g. BC1, BC5, BC7, ...).
    • Uncompressed (Ligthmaps): All layers of this tile set will use GPU caches with uncompressed texture formats (R8B8B8A8, ....) for the highest visual quality. This may increase transcoding performance and visual quality but has a very large cost in cache efficiency. Depending on the situation you may need a cache 4 to 6 times bigger to get the same efficiency.

Working Iteratively

It is common to update/add assets to a tilset using Tile Set Studio. To view the updated tile set in Unreal you should take the following steps. In the future we will try to simplify this workflow even further by allowing you to trigger studio rebuilds from within the Unreal editor.

  1. Rebuild your file in studio.
  2. Re-import in Unreal. There are two options for this currently: a. Your tile sets are stored outside your project folder: Just right click the tile set asset in the Unreal project browser and pick the 'Reimport' option in the context menu. b. Your tile sets are stored in your project's 'Content' folder: Unreal will automatically pick up the changes to the .gts file and trigger a reimport.

Gts File Management and Handling

Unlike most Unreal assets which are converted to an Unreal specific format during import, Granite still needs access to the original imported gts files to stream from during run-time. This also means the .gts files need to be shipped together with the game. This section goes in a bit more detail how Granite for Unreal handles this requirement. As of Unreal 4.11 .gts file handling is largely behind the scenes but it may still be informative for advanced uses to know how things work.

Granite handles .gts files by having a "hidden" copy of every referenced .gts file in the Content\GraniteSDK\ImportedTileSets\ folder in your project. The .gts files in this folder are fully managed by Granite, the user should in theory never touch them. When tile sets are added (through importing .gts files) or deleted (by deleting the tileset asset in the editor) Granite will automatically be sure to copy/delete the corresponding .gts files. Reimporting changed assets will correspondingly update the hidden .gts copy with the latest version. When you cook a project using Granite it will also take care that any referenced .gts files are part of the cooked build and/or copied over to the target device.

Backwards compatibility

Before Unreal 4.11 granite used .gts files stored anywhere in the Content folder leaving the user responsible for updating and deleting them and making sure they were copied to the the cooked build by adding all folders containting gts files to the Granite settings window. Granite is still backwards compatible with this workflow and will offer an upgrade path to the new workflow. This can be accessed by opening your tile sets in the Ureal editor and following the on-screen upgrade instructions (see figure below). We higly recommend upgrading your workflow as it greatly simplifies things.

Use the `Fix It` button to upgrade to the new tileset handling workflow.
