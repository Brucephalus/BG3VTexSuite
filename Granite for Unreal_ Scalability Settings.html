Scalability Settings

Granite for Unreal adds a new category to the engine scalability settings. These allow you to tweak Granite for different target hardware and usage scenarios. This page assumes familiarity with the Unreal 4 scalability system. For more info on the scalability in Unreal 4 please consult this page.

You can preview the scalability settings from within the editor.

Cache Settings

The settings corresponding to the different quality tiers are stored in "UnrealEngine/Engine/Config/BaseScalability.ini" (again, see this page for info). The following specific ini settings are made available by Granite. Note that as required by the Unreal scalability system these are in fact also console variables and can be set through other means than the scalability tiers.

r.GraniteSDK.MipBias

Values: -1 ... x, higher is faster, default 0.0

Control the relative texture resolution. See Level of Detail for more info.

r.GraniteSDK.MaxAnisotropy

Values: 0 ... 8, higher is slower, default 4

Control the rate of anisotropic texture filtering (see this page if you are unfamiliar with anisotropic filtering). In addition to reduced gpu shader impact, this will also stream significantly less data (the exact gains are less clear than with the MipBias setting but gains up to 1.5 ~ 2x in the amount of data streamed are not uncommon).

r.GraniteSDK.GPUCacheSizeScale

Values: 0.1 ... x, higher requires more GPU memory, default 1.0

Control the size of the caches relative to the sizes configured in project settings. The Project Settings window lets you configure the cache sizes for the high/epic quality level. By changing this variable you can allocate less cache memory on machines with less GPU memory. So if the project specifies 1024MB of caches and the scale is set to 0.5 for the medium tier, 512MB will be allocated on medium tier machines in practice.

Note
If you lower this settings you most likely also have to increase the MipBias accordingly or data may be streamed in with significant delays.

r.GraniteSDK.CPUCacheSizeScale

Same as GPUCacheSizeScale but for the CPU memory caches.

Note
If you change this setting without changing any of the other settings this will lead to significantly increased disc IO and by consequence increased streaming delays.

r.GraniteSDK.MinGPUCacheSizeInMB

Values: 1 ... x, higher requires more GPU memory, default 10

When working with smaller scales some of caches which were already configured to be small in the project sizes can get unreasonably small. To avoid this a minimum cache size can be configured. If caches would be smaller than this size after scaling the minimum size will be used instead.

r.GraniteSDK.MinCPUCacheSizeInMB

Same as MinGPUCacheSizeInMB but for the CPU memory caches.
